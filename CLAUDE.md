# hd-tea — Project Instructions

## What this is
hd-tea is a static HTML/CSS/JS atlas of tea, built for a tea master and their clients: origins and terroir, the six tea types and tisanes, craft from garden to firing, and the table where it lands. It shares the jiaoluo-shuwu architecture (markdown content compiled by a Node build, token-driven CSS, validators as a quality gate) but is a distinct visual and factual world.

Run locally:
```
npm run build        # generate pages/ + data/ from content/
npm run serve        # python3 -m http.server 8080  (build first)
```

Deploys to GitHub Pages as a PWA via `.github/workflows/deploy.yml` (same pattern as jiaoshu: verify → build → upload-pages-artifact → deploy).

## The design system: two registers, one set of atoms

Light mode is **The Six Liquors** (warm porcelain, pine ink, the real liquor spectrum of the six tea types as the only saturated color). Dark mode is **The Night Kiln** (Jian-ware iron glaze, kiln-flame amber, oil-spot blue). The toggle in the topnav switches between them; the choice persists in `localStorage('hd-tea-theme')` and is applied pre-paint by an inline script in the layout head.

Both registers share two structural atoms, and color is **data** in both:

- **The cup** (`.cup`): a top-down liquor circle. Every tea/tisane authors one `liquor: "#hex"` (the brewed mid-tone); `build/lib/liquor.mjs` derives the highlight and depth for the radial gradient. The cup is the hero marker, the map marker, the hub-row bullet, the homepage flight. One hex drives all of them, so two teas with the same liquor always render identically.
- **The steep curve** (`.steep-curve`): the infusion timeline, one dot per steep, shaded from the tea's liquor and rising to peak then fading. Rendered by `build/lib/steep-curve.mjs` from `steeps:` frontmatter at the `<!--STEEP_CURVE-->` marker.

Tokens live in `tokens.css` (both registers, plus the shared liquor spectrum). Components live in `style.css`. Never hardcode a color that has a token.

Typefaces: **Fraunces** (display), **Albert Sans** (body/labels), **Fragment Mono** (data only: temperatures, ratios, times, coordinates), **Noto Serif SC** (CJK). Mono is reserved for numeric data on purpose; do not use it for eyebrow labels.

## Content pipeline

Source of truth is `content/<category>/<slug>.md` (YAML frontmatter + HTML body). Never hand-edit `pages/` — it is fully regenerated by `npm run build`. Categories: `regions`, `teas`, `tisanes`, `cultivation`, `harvest`, `processing`, `storage`, `brewing`, `tasting`, `teaware`, `hubs`.

Four hubs group the categories: **Library** (teas, tisanes), **Origins** (regions), **Craft** (cultivation, harvest, processing, storage), **Table** (brewing, tasting, teaware). Membership is defined in `data/category-meta.json` (single source of truth for the build and future runtime).

Body markers the build replaces: `<!--STEEP_CURVE-->`, `<!--TERROIR_MAP-->`, `<!--HUB_MEMBERS-->`, `<!--WORLD_MAP-->`.

Full authoring spec: `templates/_drafting/page-spec.md`.

## The quality gate (this is the point)

hd-tea must never read as AI slop and must never publish an unsourced or hallucinated fact. `npm run verify` runs three validators; the pre-push hook and CI both run them.

### Voice (`build/validate-voice.mjs`)
Ported from jiaoshu's formatting validator plus the vendored humanizer ruleset.
- **ERROR (blocks build/push):** any em-dash (—) in prose or frontmatter text; double-hyphen dashes; hard health claims (cure/treat/prevent/detox/clinically proven) on teas/tisanes.
- **WARN:** AI-tell vocabulary (significance inflation, promotional language, copula avoidance, vague attribution, signposting, chatbot closers, negative parallelism, false ranges), spaced en-dash, soft health claims, slow openings, missing section anchors.

The humanizer skill is vendored verbatim at `templates/_drafting/humanizer-SKILL.md`. Run the humanizer process (draft → "what makes this obviously AI?" → rewrite) on every page; the validator only catches the machine-checkable subset. Voice rules in prose: `templates/_drafting/VOICE.md`.

### Facts (`build/validate-claims.mjs`)
- **Sources gate:** `content_review: verified` requires `content_sources` whose labels are registered in `data/_reference/known-sources.json` (unknown labels WARN). Regions and tisanes need ≥2 sources (ERROR otherwise).
- **Canonical claims:** names in `data/_reference/canonical-claims.json` cited near a year must match (ERROR; WARN when softened with "c."). Add any fact used on ≥2 pages to this registry so drift becomes a build error.
- **Cross-page consistency:** a name cited with a year range must carry the same range everywhere (ERROR).

Review workflow and checklist: `templates/_drafting/factual-review.md`. All content ships `content_review: pending` until a human works the checklist; nothing goes on the homepage or into client material until `verified`.

### Tisanes: no medical claims
Tisanes describe flavor, craft, and tradition only. Traditional-use statements cite Commission E or EMA monographs as statements about *status*, never endorsements. No dosage guidance, ever.

## Maps and figures: accurate, not decorative

Terroir maps are real. `build/lib/terroir-map.mjs` renders SVG at build time from **vendored Natural Earth 50m data** committed under `data/_reference/geo/` (public domain, checksummed in `MANIFEST.json`, no network at build time). Region frontmatter authors real coordinates:

```yaml
map:
  center: [117.98, 27.71]   # lon, lat — a wrong value is a visible bug
  span_km: 60
  markers:
    - { name: "Zhengyan core 正岩", lon: 117.96, lat: 27.70, primary: true }
```

Each region map is a Mercator detail window with a graticule and degree labels, country/province borders, rivers and lakes, a scale bar, markers at true coordinates, and a country-scale locator inset. The Origins hub carries a Natural Earth world map (`<!--WORLD_MAP-->`) marking every region. Maps are styled entirely through themed CSS classes, so one SVG serves both registers.

**Roadmap:** DEM-derived elevation contours are the next map layer. The frontmatter interface will not change when they land. Refresh vendored geo data by re-downloading from `github.com/nvkelso/natural-earth-vector` and updating `MANIFEST.json` checksums.

Diagrams (e.g. the oxidation scale on `processing/oxidation.md`) are authored as inline SVG in the body using `.figure` / `.fig-*` classes, themed through tokens. Keep them schematic and honest; label what a diagram simplifies.

## Everything is responsive and theme-aware by default
Breakpoints: 860px (sidebar → mobile TOC toggle), 700px (flight → 4 columns), 560px (steep track and other grids reflow). Wide content (maps, tables, figures) sits in `overflow-x: auto` wrappers so the page body never scrolls sideways. Both themes get equal care; `prefers-reduced-motion` is respected.

## Adding content

**A new tea:** create `content/teas/<slug>.md` with `type: tea`, `tea_type`, `liquor` (#hex), `origin` (region slug), and `steeps`. Author the body with section anchors. `npm run verify`.

**A new region:** create `content/regions/<slug>.md` with `type: region`, `country`, and a `map` block with real `center` and `markers`. Verify the coordinates like any other fact.

**A new category:** add the folder, add the slug to `entry.schema.json` → `category.enum`, add it to `data/category-meta.json` (`categories` + the right hub's `members`), and add a `--cat-<slug>` token in `tokens.css` (both registers).

**Promote a page to complete:** work `factual-review.md`, populate `content_sources` from the registry, set `content_review: verified` and `updated: today`, rebuild.
